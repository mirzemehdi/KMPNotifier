# CocoaPods → SwiftPM Migration Report

Migration of KMPNotifier's iOS Firebase integration from the CocoaPods Gradle plugin
(`kotlin("native.cocoapods")`) to Swift Package Manager (`swiftPMDependencies` DSL),
following the [Kotlin cocoapods-to-spm migration guide](https://github.com/Kotlin/kotlin-agent-skills/tree/main/skills/kotlin-tooling-cocoapods-spm-migration).

## Pre-Migration State

- **Kotlin**: 2.2.21 → required upgrade (SwiftPM import support ships since 2.4.0-Beta2; stable 2.4.0 used).
- **Modules using CocoaPods**:
  - `:kmpnotifier-push-firebase` — `pod("FirebaseMessaging")` (no version pinned), `noPodspec()`,
    framework `KMPNotifierPushFirebase` (static). Cinterop used directly:
    `cocoapods.FirebaseMessaging.FIRMessaging`, `cocoapods.FirebaseMessaging.FIRMessagingDelegateProtocol`
    in `src/iosMain/.../firebase/FirebasePushNotifierImpl.kt`.
  - `:kmpnotifier` (umbrella) — `pod("FirebaseMessaging") { linkOnly = true }` (linkage for its
    test binary; cinterop came transitively from the push module), `noPodspec()`, framework `KMPNotifier` (static).
- **iOS app (`iosApp/`)**: no Podfile/workspace — already consumed the `sample` framework via the
  `embedAndSignAppleFrameworkForXcode` build phase and already declared `firebase-ios-sdk` via SPM
  (exact 12.1.0, product `FirebaseMessaging`). No CocoaPods deintegration was needed.
- **Generated podspec**: `kmpnotifier/kmpnotifier.podspec` (checked in, generated by the plugin).
- iOS deployment target: 15.4.

## Migration Steps

1. **Kotlin upgrade**: `kotlin = "2.2.21"` → `"2.4.0"` in `gradle/libs.versions.toml`.
   Required `./gradlew kotlinUpgradeYarnLock` (js lock file churn). API dumps unchanged
   (`apiCheck` green without re-dump).
2. **`swiftPMDependencies` in `:kmpnotifier-push-firebase`** (the only module using cinterop):

   ```kotlin
   swiftPMDependencies {
       iosMinimumDeploymentTarget = "16.0"
       discoverClangModulesImplicitly = false
       swiftPackage(
           url = url("https://github.com/firebase/firebase-ios-sdk.git"),
           version = exact("12.1.0"),           // matches the version pinned in iosApp
           products = listOf(product("FirebaseMessaging")),
           importedClangModules = listOf("FirebaseMessaging"),
       )
   }
   ```

   `discoverClangModulesImplicitly = false` per guide: Firebase's transitive C/C++ modules
   (gRPC, abseil, …) fail cinterop generation.
3. **Import transformation** (declaring module's namespace — group `io.github.mirzemehdi`,
   module `kmpnotifier-push-firebase`, dashes → dots):

   ```kotlin
   // BEFORE
   import cocoapods.FirebaseMessaging.FIRMessaging
   import cocoapods.FirebaseMessaging.FIRMessagingDelegateProtocol
   // AFTER
   import swiftPMImport.io.github.mirzemehdi.kmpnotifier.push.firebase.FIRMessaging
   import swiftPMImport.io.github.mirzemehdi.kmpnotifier.push.firebase.FIRMessagingDelegateProtocol
   ```
4. **CocoaPods removal**: dropped `kotlinNativeCocoaPods` plugin from `:kmpnotifier-push-firebase`,
   `:kmpnotifier`, the root plugins block and `libs.versions.toml`; deleted both `cocoapods {}`
   blocks and the generated `kmpnotifier/kmpnotifier.podspec`. The umbrella's `linkOnly` pod
   workaround (added earlier in 2.0 development to avoid duplicate cinterop symbols) is obsolete —
   consuming modules need no SPM declaration; linkage of test binaries is handled by the
   declaring module's `swiftPMDependencies`.
5. **Framework blocks dropped, not moved**: the cocoapods `framework {}` blocks existed only
   because the CocoaPods plugin requires them. The published artifacts are klibs; the sample app
   exports the modules into its own framework (`sample/build.gradle.kts`), so no
   `binaries.framework {}` replacement was added to library modules.
6. **Xcode**: no project changes needed. The Gradle build phase (`embedAndSignAppleFrameworkForXcode`)
   and the app's own `firebase-ios-sdk` SPM dependency were already in place.
   `ENABLE_USER_SCRIPT_SANDBOXING` is absent from the project (Xcode default YES) — the verification
   build passes it as `NO` on the command line; if Xcode-GUI builds hit sandbox errors, set it to NO
   in the project build settings.

## Import Transformations

| Before | After | Note |
|---|---|---|
| `cocoapods.FirebaseMessaging.FIRMessaging` | `swiftPMImport.io.github.mirzemehdi.kmpnotifier.push.firebase.FIRMessaging` | push module iosMain |
| `cocoapods.FirebaseMessaging.FIRMessagingDelegateProtocol` | `swiftPMImport.io.github.mirzemehdi.kmpnotifier.push.firebase.FIRMessagingDelegateProtocol` | push module iosMain |

No preserved `cocoapods.*` imports remain anywhere in the repository.

> **Consumer note:** published 1.x klibs exposed the `cocoapods.FirebaseMessaging` namespace
> (KMPNotifier is listed in the migration guide as a library with bundled cocoapods klibs).
> From 2.0.0 the bundled namespace is `swiftPMImport.io.github.mirzemehdi.kmpnotifier.push.firebase`.
> Apps that imported `cocoapods.FirebaseMessaging.*` *through* KMPNotifier's klib must update those
> imports or declare their own Firebase dependency.

## Errors Encountered

- **Error #1** — Phase 7 (pre-migration, CocoaPods era): umbrella test binary failed with
  `Linking globals named 'knifunptr_cocoapods_FirebaseMessaging1_...': symbol multiply defined!`
  when both the umbrella and the push module declared `pod("FirebaseMessaging")`.
  Root cause: two cinterop bindings of the same pod on one link path. Interim fix was
  `linkOnly = true`; the SPM migration removes the problem entirely (single declaring module).
  Generalizable: yes — never declare the same pod's cinterop in two modules of one dependency graph.
- **Error #2** — Phase 2: `kotlinStoreYarnLock` failed after the Kotlin bump.
  Fix: `./gradlew kotlinUpgradeYarnLock`. Generalizable: yes — expected on any Kotlin upgrade
  with js targets.

## Non-Trivial Decisions

- **iOS minimum deployment target raised 15.4 → 16.0** (guide recommendation for SwiftPM import).
  Documented in CHANGELOG/README/MIGRATION.
- **Firebase version pinned `exact("12.1.0")`** to match the version already used by `iosApp`,
  per the guide's version-preservation rule (the pod had no explicit version).
- **Static linking preserved** (frameworks produced by the sample remain static).
- **Kotlin 2.4.0 (stable)** chosen over 2.4.0-Beta2 — released 2026-06-03, first stable line with
  `swiftPMDependencies`.
